# 入料 - 物料类型

# 背景

低代码工作台，本质上是对素材的组装，工作台的物料即组件。素材有基础组件，也有复合组件。基础组件是指直接代码编写的 pro-code 组件，复合组件是将基础组件通过一个 schema 片段组合而成的，也叫区块/模板。

# 基础组件

基础组件就是我们平时常见的 react 组件，在可视化编辑器里可以被作为最小单元排列组合。在渲染引擎中，这些基础组件就是递归渲染的每个节点的组件；而对于编辑器而言，需要的是它的描述信息，比如在物料面板中展示的图标、名称、在属性设置面板展示的配置参数，这份参数大概要包含的信息如下：

{
    "componentName": "Page",
    "title": "Page",
    "desc": "",
    "props": [
      {
        "propType": "string",
        "name": "font",
        "required": true,
        "isRequired": true,
        "description": "字体库",
        "setter": "FontSetter"
      },
      {
        "propType": "Object",
        "name": "style",
        "required": true,
        "isRequired": true,
        "description": "主题配色",
        "setter": "ColorThemeSetter",
        "children": [
          {
            "propType": "string",
            "name": "color",
            "isRequired": true,
            "description": ""
          },
          {
            "propType": "number",
            "name": "type",
            "isRequired": true,
            "description": ""
          }
        ]
      }
    ]
  }

组件描述信息，在 lowcode-engine (opens new window) 有一份特别大的物料协议，包括组件名、封面图、参数列表、参数类型、参数在属性面板的 setter 。然而这么复杂的信息都需要写 json ，而且每次修改组件参数都要手动改一次这个 json，似乎并不是特别美妙的一件事。在 TS抽取组件描述 (opens new window) 中有详细分析中有详细介绍，有哪些生成 json 描述的方式，tsx 抽取类型声明直接生成组件描述是其中一种方式：一份卡片代码，会被构建脚本编译成两份产物，一份是卡片描述信息（卡片元信息）、一份是给渲染用的真实 UMD 代码。元信息是给编辑器用于被拖拽和参数配置用的， UMD 代码用于给渲染引擎执行用。

# 复合组件

复合组件，其实并非一个真实存在的组件，只是一个 schema 片段，在设计器上被拖入画布时，本质上就是将这个树枝片段置入到整个 schema 树里，结构类似于：

{
  componentName: "Cards"
  id: "node_2298a86a3406"
  nodeAlias: "卡片#01"
  props: {layout: "vertical", title: "卡片标题"}
	allowedChildType: ["Text","Button"],
	children: [{
    componentName: "Text"
    id: "node_2298a86a3407"
    nodeAlias: "卡片#01"
    props: {color: "red", text: "一段文本"},
    allowedDelete: false
  },{
    componentName: "Button"
    id: "node_2298a86a3408"
    nodeAlias: "卡片#01"
    props: {color: "red", text: "按钮"}
    allowedDelete: false
  }]
}

看起来跟页面的 Schema 很相似，实际上是一段已经初始化了各项参数的组合，而页面本身也是递归组合的组件节点树描述。特殊的是，Card 节点限制了可以被拖入的组件类型、其子节点限制了不允许删除。这些信息是基础组件描述之上的拓展，可以被插件识别，用于限制拖入非法节点或者新节点、禁止删除节点。从而使得整个复合组件看起来是一个大颗粒组件整体，而不是一个个零件。

相对于基础组件可以通过手写 / TS 识别，复合组件的组件描述显然复杂得多，因为它实际上已经是从 O(1) 到 O(n) ，甚至如果组件之间有联动，这一结构编写的复杂度甚至会到达 O(n2) , 如何生产复合组件，就成了一个困难的问题。 可以直接手写；可以通过一个可视化编辑器，专门用于生产复合组件，里面支持自由布局、定义入参、定义样式和内置行为逻辑，本质上是对复合组件手写代码的一个可视化升级；可以通过编写 TSX 组件通过语法树构建；可以通过 TSX 运行时 fiber 树分析，这些方法里面，可能就可视化配置会比较高效率一些，所以产生了阿里低代码的“造物”系统，通过可视化编辑器的方式，以配置的方式生产出来复合型组件。
